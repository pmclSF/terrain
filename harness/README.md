# Terrain validation harness

The harness keeps the LB quality bars honest across releases. It runs on every release and produces per-rule readiness cards that ship with the release tag.

Full design in [`docs/HARNESS.md`](../docs/HARNESS.md). This README is the entry point for maintainers running the harness locally.

## Directory structure

```
harness/
├── README.md            ← you are here
├── triage-panel.md      ← recruitment, compensation, protocol (TODO: populate before first panel run)
├── corpora/             ← hand-labeled PR corpora per dogfood repo
│   ├── terrain-testing-fullstack-rag/labels.yaml
│   ├── terrain-testing-go-monolith/labels.yaml
│   ├── terrain-testing-ai-only/labels.yaml
│   ├── terrain-testing-polyglot-monorepo/labels.yaml
│   └── terrain-testing-ml-pipeline/labels.yaml
├── runner/              ← Go scripts that invoke Terrain on each labeled PR
├── validators/          ← automated LB checks (schema, silence-on-green, reproduction parity, FP rate, recall, JUnit renderer, runtime, fail-mode, bidirectional attribution)
├── readiness/           ← per-rule readiness cards per release
│   └── v0.2.0/          ← cards committed alongside the v0.2.0 release tag
└── reports/             ← raw runner outputs (gitignored except for the most recent)
```

## What's here at 0.2.0 development time

This skeleton is the structural placeholder. Each subdirectory contains a `README.md` describing what fills it. Implementations land per the Tier 4 task in the parent project's task list.

## How to run the harness locally (once implemented)

```bash
cd harness/
go build -o run-harness ./runner
./run-harness --release v0.2.0 --repos all --validators all
./run-harness --release v0.2.0 --report  # produces readiness cards
```
