# Terrain 0.2 Maturity Audit

**Date:** 2026-05-02
**Branch:** `docs/0.2.x-maturity-audit`
**Audit subject:** `main` at `8545f03` (post PR #131 merge; PRs #132/#133/#134 open)

## Why this exists

A senior-engineer launch-readiness review (2026-05-02) flagged that Terrain
0.2's product surface is "too wide" and recommended cutting scope. We
reject that premise: the product use case is **aligning testing across
multiple repos** — including monorepo migration toward a single framework
mix and the broader job of turning a fragmented test base into a healthier
one. Conversion is a *tool that serves alignment*, not the headline. The
right response to "uneven product surface" is to lift the floor across
all functional areas, not to drop areas to make the average look better.

This document is the audit that operationalizes that response. Twelve
functional areas × fourteen maturity axes. Each cell scored 1–5 with
evidence. The deliverable is:

1. A current-state floor map (per-area lowest score)
2. A horizontal pattern analysis (per-axis weakest areas)
3. A "uniform mature" target (proposed: 3 across the board for 0.2.x
   stable; 4 where confidence claims are made externally)
4. A backlog ordered by floor-first then horizontal-first

The audit doesn't change behavior. Code/UX changes that follow from it
are tracked as separate PRs.

---

## Rubric

Two halves: **product** (what users see) and **engineering** (what
reviewers find under the hood). Each axis 1–5; the *floor* matters more
than the average — a senior evaluator judges by the weakest link.

### Product axes

| # | Axis | 1 (raw) | 3 (workable) | 5 (mature) |
|---|------|---------|--------------|------------|
| P1 | Feature completeness | Skeleton; common flows missing | Common flows work end-to-end | Every documented flow works including edge cases |
| P2 | Output trust | Anecdotal/heuristic only | Calibrated on synthetic fixtures | Calibrated against labeled real-repo corpus with published precision/recall |
| P3 | Documentation | README mentions exist | Per-feature doc with caveats | Worked example + known false positives + suppression guidance |
| P4 | Onboarding | Requires reading source | Quickstart works on a sample repo | Quickstart works on an unfamiliar real repo in <15 min |
| P5 | Error UX | Stack traces / opaque | Actionable but inconsistent | Every error has a remediation pointer |
| P6 | Examples | One in README | Examples per command | Reproducible per-platform examples in CI |
| P7 | Use-case fit | Solves a fraction of the user's job | Solves the headline job | Solves the job + composes with neighbouring tools |

### Engineering axes

| # | Axis | 1 (raw) | 3 (workable) | 5 (mature) |
|---|------|---------|--------------|------------|
| E1 | Test coverage | Unit only, sparse | Unit + integration | Unit + integration + golden + property/fuzz |
| E2 | Calibration | None | Recall-anchor fixtures | Precision floor against real-repo corpus |
| E3 | Observability | None | Logs on demand | Per-component timing + completeness score in normal output |
| E4 | Stability | Schema unversioned | Schema versioned, fields can move | Stable/beta/internal field tiers documented |
| E5 | Performance | Unbounded | Walks complete on Terrain self-scan | Memory + time budgets documented per repo size class |
| E6 | Determinism | None | `SOURCE_DATE_EPOCH` honored | Byte-identical CI gate |
| E7 | Cancellation | `context.Background()` | Context plumbed at top level | Context honored in inner loops; tested with deliberately-slow detectors |

**Target line for 0.2.x stable:** every cell ≥ 3.
**Target line for "ready to claim publicly":** every cell ≥ 4 in any
area where the area is named in marketing.

## Functional taxonomy

| # | Area | Primary surface | Why it's separate |
|---|------|-----------------|-------------------|
| 1 | Core analyze pipeline | `analyze`, snapshot, signal manifest | The foundation every other area builds on |
| 2 | Insights / impact / explain | `report insights/impact/explain` | Read-side queries over the snapshot |
| 3 | Summary / posture / metrics / focus | `report summary/posture/metrics/focus` | Aggregation views over signals |
| 4 | PR / change-scoped | `report pr`, `report select-tests`, `compare` | Workflow-scoped, distinct trust requirements |
| 5 | AI risk + inventory | `aiSurface*` detectors, surface inference | Heuristic; needs separate trust framing |
| 6 | AI eval ingestion | Promptfoo / DeepEval / Ragas adapters | Different code path; integration-testing target |
| 7 | AI execution + gating | `terrain ai run` (direct execute + dry-run), `ai gate` (planned) | Sandbox + secrets concerns |
| 8 | Migration / conversion | `migrate`, `convert`, framework-pair converters | Used in service of repo alignment, not for its own sake |
| 9 | Portfolio | `terrain portfolio` | Multi-repo aggregation; cross-repo alignment story |
| 10 | Policy / governance | `policy check`, ownership | Operational risk dimension |
| 11 | Server | `terrain serve` | Different concurrency/security profile |
| 12 | Distribution / install | npm / brew / cosign / postinstall | Pre-first-run trust surface |

**Why no separate "config / preferences" area?** `terrain config
{feedback,telemetry}` is a thin wrapper; covered under area 12 (install)
and area 1 (analyze) since the consequential settings are pipeline-side.

**Why no separate "debug" area?** `terrain debug *` is a diagnostic
view *over* the analyze pipeline; covered under area 1.

## How to read the score tables

- **5** = production-grade, evidence cited, would survive a hostile
  review.
- **4** = solid foundation + one reachable gap. Public claims OK.
- **3** = workable for early adopters; private use; no marketing
  claims beyond "experimental".
- **2** = ships but visibly incomplete; expect bug reports.
- **1** = code path exists but isn't trustworthy.

Each cell carries a 1-line evidence pointer (file:line, test name,
or "no evidence found"). Where the score is contestable, evidence
makes the audit reviewable.

**To lift to floor=4:** publish-quality mid-pipeline error messages with
remediation; add per-detector timing to verbose output; write a
deliberately-slow-detector test that proves `ctx.Err()` is honored
inside parser walks.

## Proposed uniform-target for "publicly claimable" capabilities

**Public-claim target: every cell ≥ 4 in any area named in marketing.**

Today, the README names every functional area. To honor the public-
claim target, either:

- **Path A:** Lift every area to floor ≥ 4 — large body of work,
  multi-quarter.
- **Path B:** Distinguish "named in marketing" (must be ≥ 4) from
  "available but not headline" (≥ 3) — a tier split in the README.

implies what we can defend.

  Area 8 (migration/conversion *with alignment framing*),

  Area 10 (policy), Area 11 (server).

the user wants to defend stays, but with honest labels.
