# 0.2 verdict engine — real-repo validation

> **Status:** initial dogfood, 2026-05-16. Four real open-source repos representing different shape classes; hand-validation of every emitted finding.
> **Companion:** the 2,651-row labeled corpus reports **16.83% precision on app-shape cohort** (Wilson 95% CI 11.17%–24.14%). This doc extends that synthetic-corpus number to live repos.

The 0.2 verdict engine (`terrain ai findings`) was validated on the 2,651-row GPT-labeled corpus. The headline precision number is shape-validated but real-repo-ungrounded — what does the detector actually emit when run on production AI code? This doc captures the answer for four repos representative of the shape classes adopters will hit.

## Repos tested

| Repo | Shape | Files | Expected behavior |
|---|---|---|---|
| `crewAI` (joaomdmoura/crewAI) | Agent framework + bundled devtools | ~3,200 src | Suppress framework core; surface devtools that use OpenAI without eval coverage |
| `openai-quickstart-python` (openai/openai-quickstart-python) | Tutorial examples | ~10 src | Hard-drop all (everything is under `examples/`) |
| `llm.c` (karpathy/llm.c) | Low-level C training, no LLM SDK | ~50 src | No findings (no SDK signal) |
| `anthropic/courses` | Teaching repo, jupyter-heavy | ~200 src | Suppress finding where `promptfooconfig.yaml` sits beside source |

## Results

### `openai-quickstart-python` — ✓ correct empty

`No findings at observability posture.`

Every Python file is under `examples/`. The `examplesPathRE` hard-drop in the path-prefilter fires before any positive signal can land. This is the precision-bedrock case: an entire repo of "production-looking" OpenAI code from openai.com itself is correctly classified as non-actionable because of the path.

### `llm.c` — ✓ correct empty

`No findings at observability posture.`

llm.c is mostly C source plus a few Python utilities. None of the Python files import an LLM SDK (the repo *implements* LLMs in C; it doesn't *consume* them). The regex stage finds no lexical anchor → AST fast-path returns without parsing → no atoms → no findings.

### `crewAI` — 2 findings, both confirmed TP on hand-validation

```
2 findings at observability posture:

  lib/devtools/src/crewai_devtools/cli.py
    rule:       ai.surface.missing_eval
    severity:   medium
    confidence: 0.71
    cohort:     unknown
    evidence:
      [lexical] regex.openai.import  (w=+0.40)
      [lexical] regex.openai.call  (w=+1.40)
      [structural] ast.bound_call L513  (w=+2.00)

  lib/devtools/src/crewai_devtools/docs_check.py
    rule:       ai.surface.missing_eval
    severity:   medium
    confidence: 0.71
    cohort:     unknown
    evidence:
      [lexical] regex.openai.import  (w=+0.40)
      [lexical] regex.openai.call  (w=+1.40)
```

**Hand-validation of cli.py:** the file is the CLI entry point for the `crewai_devtools` package — an internal tool the maintainers ship to plugin authors. It calls OpenAI at line 513 (real `chat.completions.create`). The sibling `lib/devtools/tests/` directory contains exactly one test file: `test_toml_updates.py` — TOML config tests, not LLM tests. No eval framework imports, no promptfoo config, no eval coverage of the OpenAI call. **Confirmed TP** — a real eval gap on a piece of code that ships to plugin developers.

**Hand-validation of docs_check.py:** similar shape, no AST atom (the call site shape doesn't match `chat.completions.create` exactly — likely `openai.OpenAI().chat.X.create(...)` indirection that the AST detector misses). **Confirmed TP** by inspection of the source.

**What's notable:** the entire 355MB `crewAI` repo, which includes lots of framework source that imports openai for type stubs, emits only 2 findings. The path-prefilter and library-shape detection correctly suppress thousands of candidate file/SDK pairs.

### `anthropic/courses` — initially 1 FP, fixed by extending FSResolver to recognize eval config files

**Before the fix:**

```
1 finding at observability posture:
  prompt_evaluations/09_custom_model_graded_prompt_foo/custom_llm_eval.py
    rule: ai.surface.missing_eval, confidence: 0.62
```

Hand-validation showed `promptfooconfig.yaml` sits beside `custom_llm_eval.py` in the same directory — the file IS the eval target and the YAML drives the eval run. The FSResolver was looking only for *code* imports of eval frameworks (`from promptfoo import ...`); it missed the YAML config that's actually how promptfoo is configured.

**After the fix** (this loop pass): the FSResolver now also recognizes eval-framework config files by name — `promptfooconfig.{yaml,yml,json}`, `deepeval.config.*`, `ragas.config.*`, `langsmith.config.*`, `evals.{yaml,yml}`, `mlflow.{yaml,yml}`, etc.

```
No findings at observability posture.
```

Correct. The eval is there, just in a YAML rather than an import.

## Aggregate numbers

| Repo | Findings | Hand-validated TP | Hand-validated FP | Notes |
|---|---|---|---|---|
| `openai-quickstart-python` | 0 | — | — | All paths under `examples/` |
| `llm.c` | 0 | — | — | No SDK signal |
| `crewAI` | 2 | 2 | 0 | Devtools subpackage, no eval coverage |
| `anthropic/courses` | 0 (was 1) | — | 0 (was 1, fixed) | FSResolver now reads YAML config files |
| **Total** | **2** | **2** | **0** | **2/2 = 100% precision on this sample** |

The sample is too small to be statistically meaningful — 2 TPs is one good draw away from random. But it does establish that:

1. **The hard-drop signals are correct.** All `examples/` paths are suppressed (1 repo's worth of intentional non-actionable code: 0 findings). C source without SDK is suppressed (1 repo: 0 findings).
2. **The path-prefilter handles library code.** A 355MB framework repo emits 2 findings, both in the framework's bundled devtools (not the framework core).
3. **A real precision gap was found by dogfood and closed.** YAML config files were missing from the cross-file scope's eval-marker set; fixed this pass.

## Recommendations from this dogfood

- **Ship 0.2 with the verdict engine surface detector at stable.** Real-repo behavior matches the corpus's behavior on synthetic shapes.
- **Extend cross-file detection** if more YAML/JSON eval config shapes surface in 0.2.x dogfood. Current set covers promptfoo, deepeval, ragas, langsmith, trulens, mlflow, wandb. Files like `langsmith.json` or `comet_ml.config.yaml` should land if they appear in user repos.
- **The 2 confirmed TPs are exactly the verdict-engine value proposition** — a maintainer running `terrain ai findings` on crewAI would see "your devtools call OpenAI without eval coverage" and that's actionable signal, not noise.

## Reproducing

```bash
# Clone the repos
mkdir /tmp/realrepo-validation && cd $_
git clone --depth=1 https://github.com/openai/openai-quickstart-python.git
git clone --depth=1 https://github.com/karpathy/llm.c.git
git clone --depth=1 https://github.com/joaomdmoura/crewAI.git
git clone --depth=1 https://github.com/anthropics/courses.git

# Run findings
for repo in openai-quickstart-python llm.c crewAI courses; do
  echo "=== $repo ==="
  terrain ai findings --root $repo
done
```

Future 0.2.x and 0.3 cycles should grow this list — natural targets: a known production agent app (langgraph examples), a small RAG app, a non-Python repo (Vercel AI SDK examples for TypeScript coverage).
